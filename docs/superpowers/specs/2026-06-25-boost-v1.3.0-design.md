## Section 1: Security & Stability

### 1a. CSRF protection on web server
`boost-web.py` runs as root, accepts POST requests without origin validation.  
**Fix:** All POST handlers check `Origin` header == `http://127.0.0.1:8765`. Reject with 403 otherwise. No external deps.

### 1b. AMD RAPL path detection
`apply_hardware_limits()` and `record_stats()` hardcode `/sys/class/powercap/intel-rapl/intel-rapl:0/...`. Missing on AMD systems.  
**Fix:** `detect_rapl_base()` function tries `intel-rapl:0` first, then AMD hwmon `power1_average` fallback. Used in `power-common.sh` and daemon.

### 1c. Atomic state file writes
`SNOOZE_FILE` and `SKIP_TODAY_FILE` written with `echo` / `open(f, 'w')`. Race condition between daemon and web server.  
**Fix:** Write to `.tmp` then `os.rename()` (atomic on Linux) in daemon and web server.

### 1d. Stale originals protection
`save_originals()` never runs if file exists — even if hardware changed.  
**Fix:** Add GPU model check at install time. If mismatch, `--refresh` originals. Document in install output.

## Section 2: AMD GPU Support

Biggest credibility gap: README claims "Intel + AMD + NVIDIA" but GPU management is 100% NVIDIA.

**Kernel interface (amdgpu driver):**
- `/sys/class/drm/card*/device/hwmon/hwmon*/power1_cap` — power limit (µW, read/write)
- `/sys/class/drm/card*/device/hwmon/hwmon*/power1_average` — current draw (µW)
- `/sys/class/drm/card*/device/hwmon/hwmon*/temp1_input` — GPU temp (millidegrees)

**Implementation:**
- `find_amd_gpu_hwmon()` in `power-common.sh` — discovers amdgpu hwmon path
- `get_gpu_csv()` dispatches: NVIDIA first (nvidia-smi), AMD fallback, empty if neither
- `apply_hardware_limits()` gains AMD branch alongside NVIDIA
- `save_originals()` saves `ORIG_AMD_GPU_LIMIT`
- Daemon `get_gpu_stats()` same dispatch
- Web dashboard shows GPU stats vendor-agnostically

**Limits:**
- amdgpu driver must be loaded (standard on all modern AMD dGPUs)
- Writing `power1_cap` requires root (already have it)
- No AMD GPU scaling on systems with only iGPU — graceful skip

## Section 4: Process Detection + Dashboard History

### Daemon workload detection
Extend existing game detection with two new categories:

**CREATOR_PROCESSES** = `['ffmpeg', 'blender', 'HandBrakeCLI', 'kdenlive', 'davinci', 'cargo build', 'make', 'cmake', 'gcc', 'g++', 'nvcc']`
- When detected and profile != performance and temp < boost_temp_limit: suggest/switch to creator mode (same pattern as game detection but with notification)

**MEETING_PROCESSES** = `['zoom', '.zoom', 'teams', 'slack', 'discord', 'obs', 'pipewire-camera-src']`
- When detected and mode not quiet: send one-time notification "Video call detected — switch to Quiet mode?" with action button

Both auto-clear when process disappears.

### Profile switch history in dashboard
Stats CSV already has `profile` column. Dashboard telemetry chart gets:
- Vertical colored bands when profile changes (performance=red, balanced=green, power-saver=blue)
- Mini "Recent Switches" log below chart: last 5 profile changes with timestamp

## Implementation Order

1. Security fixes (CSRF, atomic writes) — no risk, pure hardening
2. AMD RAPL detection — stability for AMD users
3. AMD GPU support — biggest feature gap
4. CI/testing — validates all changes
5. Process detection — new daemon features
6. Dashboard history — web-only change
7. Repo polish — README, SECURITY.md, PR template, CHANGELOG
